# Issue #241: Rename Group→Cadence, Tag→Group — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Eliminate the backend/UI terminology gap by renaming `Group`→`Cadence` and `Tag`→`Group` throughout the entire Swift codebase, preventing the class of bugs caused by the naming mismatch.

**Architecture:** Two-phase rename to avoid naming collisions: Phase 1 renames `Group`→`Cadence` (safe, `Cadence` doesn't exist), then Phase 2 renames `Tag`→`Group` (safe after Phase 1). Core Data entity/attribute names stay unchanged — the mapping layer bridges old CD names to new domain names, avoiding any Core Data migration.

**Tech Stack:** Swift, SwiftUI, Core Data (no migration needed), Codable (JSON backward compat)

---

## Key Design Decisions

1. **Core Data entities stay unchanged**: `GroupEntity`, `TagEntity`, `PersonEntity.groupId`, `PersonEntity.tagIds` keep their current names in the `.xcdatamodeld`. The mapping extensions translate to new domain types (`Cadence`, `Group`, `Person.cadenceId`, `Person.groupIds`). This avoids Core Data migration entirely.

2. **JSON export uses new key names**: Exports produce `cadences`/`groups` instead of `groups`/`tags`. Import accepts BOTH old and new key names for backward compat.

3. **UI strings stay unchanged**: Navigation titles like "Manage Frequencies", "Manage Groups", "Change Frequency" already use user-facing language and don't change.

4. **Execution order matters**: Phase 1 (Group→Cadence) MUST complete before Phase 2 (Tag→Group) starts, or file renames will collide.

### Task 7: JSON Export/Import backward compatibility

**Context:** The Codable structs changed names and properties. Export produces new JSON format. Import must accept BOTH old format (pre-rename) and new format.

**Changes needed in ExportModels.swift:**

```swift
// ExportData — use CodingKeys for backward compat
struct ExportData: Codable {
    let version: Int
    let exportedAt: Date
    let cadences: [ExportCadence]  // was: groups
    let groups: [ExportGroup]      // was: tags
    let people: [ExportPerson]

    // Accept old format on decode
    enum CodingKeys: String, CodingKey {
        case version, exportedAt, cadences, groups, people
    }

    init(from decoder: Decoder) throws {
        let container = try decoder.container(keyedBy: CodingKeys.self)
        version = try container.decode(Int.self, forKey: .version)
        exportedAt = try container.decode(Date.self, forKey: .exportedAt)
        people = try container.decode([ExportPerson].self, forKey: .people)
        // New format: "cadences" + "groups"
        // Old format: "groups" (cadences) + "tags" (groups)
        if let c = try? container.decode([ExportCadence].self, forKey: .cadences) {
            cadences = c
            groups = (try? container.decode([ExportGroup].self, forKey: .groups)) ?? []
        } else {
            // Legacy: "groups" key contained cadences
            cadences = (try? container.decode([ExportCadence].self, forKey: .groups)) ?? []
            // Legacy: "tags" key contained groups — need a fallback key
            groups = []  // handled by legacy ExportData decoder below
        }
    }
}
```

Actually, for clean backward compat, the simplest approach: try decoding as new ExportData first. If that fails, try decoding as a LegacyExportData struct, then convert.

**ExportPerson backward compat:**
```swift
struct ExportPerson: Codable {
    let cadenceId: UUID?      // was: groupId
    let cadenceName: String?  // was: groupName
    let groupIds: [UUID]      // was: tagIds
    let groupNames: [String]  // was: tagNames
    // ... other fields unchanged

    // Custom init(from:) to accept old key names
    init(from decoder: Decoder) throws {
        let c = try decoder.container(keyedBy: CodingKeys.self)
        cadenceId = try? c.decode(UUID.self, forKey: .cadenceId)
            ?? c.decode(UUID.self, forKey: .legacyGroupId)
        // etc.
    }
}
```
